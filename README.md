# DevPilot

DevPilot is a macOS menu bar workflow assistant for developers, students, and power users.

DevPilot 是一个面向 Mac 开发者、学生和效率用户的菜单栏工作流助手，帮助你快速启动工作流、检测开发环境、运行常用命令并整理下载目录。

## Features

- Menu bar quick access.

- Custom workflows.
- Development environment checker.
- Command center for built-in safe commands.
- Downloads organizer.
- Safe shell command execution.
- Native macOS app bundle experience.

## Who It Is For

DevPilot v0.1.0 is designed for Mac developers, computer science students, and power users who want a lightweight place to start common local workflows. It is intentionally small: it does not try to replace Raycast, Alfred, full IDEs, or package managers.


Then:

1. Unzip the file.
2. Drag `DevPilot.app` to Applications.

```text
DevPilot-v0.1.0.dmg
```

Open the DMG, then drag DevPilot to Applications.

## First Launch

DevPilot v0.1.0 may not be signed with an Apple Developer ID or notarized. If macOS says the app cannot be opened because the developer cannot be verified:

1. Right-click DevPilot.app.
2. Choose Open.
3. Click Open again.

Do not disable Gatekeeper.

The release script applies a local ad-hoc signature when `codesign` is available. This helps local validation, but it is not the same as Developer ID signing or Apple notarization.

## Usage

- Launch DevPilot from Applications.
- The main window opens on launch.
- A menu bar icon stays available.
- Closing the window keeps DevPilot running.
- Use Command + Q or the menu bar Quit action to exit.

DevPilot includes these v0.1 workflows:

- Open and run custom workflows.
- Check development tools such as Git, Homebrew, Swift, Node.js, Python, Docker, Flutter, and editor CLIs.
- Run built-in command snippets with stdout, stderr, exit code, duration, and working directory display.
- Organize files in the Downloads folder by moving recognized file types into category folders. DevPilot does not delete files.
- Save settings for project folders, local URLs, command working directories, editor preference, and environment auto-checking.

## Build From Source

DevPilot is currently a Swift Package Manager macOS app.

```bash
git clone https://github.com/<your-username>/DevPilot.git
cd DevPilot
open Package.swift
```

In Xcode:

1. Select the `DevPilot` scheme.
2. Select My Mac.
3. Press Run.

You can also use the SwiftPM workspace generated by Xcode:

```bash
open .swiftpm/xcode/package.xcworkspace
```

## Architecture

- `Sources/DevPilot/App` contains app startup, lifecycle, menu bar status item, and main-window ownership.
- `Sources/DevPilot/Views` contains SwiftUI screens for home, commands, workflows, environment checks, downloads organizing, and settings.
- `Sources/DevPilotCore/Models` contains data models shared across the app.
- `Sources/DevPilotCore/Services` contains shell execution, environment checks, file organizing, workflow persistence, workflow running, and command safety validation.
- `Sources/DevPilotCore/Utilities` contains layout constants, logging, menu bar identity, and path helpers.

## Safety

The command center only runs built-in command snippets. Custom workflow shell actions are checked by `CommandSafetyValidator`, which blocks clearly dangerous commands such as destructive `rm`, disk erase, unsafe ownership changes, fork bombs, and Finder termination commands.

Commands that can modify a system or project, such as installs or privileged commands, are marked as needing confirmation. Downloads organizing moves files into folders and handles name collisions; it does not delete files.

Because macOS GUI apps do not inherit the full Terminal shell environment, DevPilot configures a predictable PATH for common developer tools and captures both stdout and stderr. This matters for tools such as `java -version` and `dart --version`, which often write version output to stderr.

## FAQ

### Why is there no `.xcodeproj`?

DevPilot is a SwiftPM app. Open `Package.swift` in Xcode and choose the `DevPilot` scheme.

### Does DevPilot require an Apple Developer account?

No for local builds and ZIP distribution. Without Developer ID signing and notarization, users may see the standard macOS first-launch warning.

### Is the local ad-hoc signature enough for public distribution?

No. Ad-hoc signing is useful for local validation. A polished public release should use Developer ID signing and Apple notarization.

### Why does Docker show installed but commands still fail?

DevPilot can detect the Docker CLI, but Docker Desktop may not be running. In that case Docker commands can still return runtime errors.

### Why is `source .venv/bin/activate` copy-only?

Activation changes the current shell session. DevPilot runs commands in independent processes, so activation would not persist for later commands.

## Known Limitations

- DevPilot v0.1.0 may not be Developer ID signed or notarized.
- Some commands depend on the user's local tools and PATH.
- Docker CLI availability does not guarantee Docker Desktop is running.
- Android Studio detection checks the app path first and may not report a precise version.
- Custom workflows are intentionally simple in v0.1.0.
- Import/export, notifications, delayed actions, clipboard actions, automatic updates, and AI features are not included yet.
